5 月号, 失敗事例の章の冒頭抜粋

2026-05 失敗事例: cache_control 空テキストブロックでセッションが完全に詰まる

何が起きるか

短いセッションでも突然、 API から 400 エラーが返り続けて作業が止まる。 代表的なメッセージは次の 2 つで、 順番にこの形で出ることが多い。

messages.N.content.M.text: cache_control cannot be set for empty text blocks
messages: text content blocks must be non-empty

「Try again」 も claude --resume も同じエラーで止まる。 新しいセッションを始めると直るが、 それまでの文脈は捨てることになる。

なぜ詰まるか

会話履歴のどこかの content block が text: "" のまま cache_control: { type: "ephemeral" } を付けた状態で残り、 API がそれを拒否する。 一度この壊れたブロックが履歴に入ると、 再送のたびに同じブロックが含まれるので、 resume も Try again も同じ理由で落ちる。

「短いセッションだから長文ではない」 と感じても、 ツールを 1 回使うごとに assistant tool_use と user tool_result の 2 メッセージが追加される。 ツール多用のセッションなら数ターンで messages[48] 程度まですぐ伸びる。

どんなときに発生しているか (2026 年 4-5 月の集積から)

  • 画像をキャプションなしで貼り付けた直後
  • ストリームの待機タイムアウト後の自動再送
  • AskUserQuestion の回答が空のまま送られたとき
  • セッション再開直後に履歴が壊れていたとき

GitHub Issue では同じ根本原因で別々のスレッドが 12 本以上 立っている。 本ガイドの公開前に各スレッドの本文を確認し、 エラー文言と再現手順が一致するものだけを残した: #55369 / #55156 / #55302 / #55283 / #55118 / #54988 / #54421 / #54314 / #53632 / #50738 / #50681 / #50010 など。

何を届けるか（毎月）

    料金、秘密情報、破壊的操作、長時間セッション、公開前確認の 5 領域の月次更新版。前月との差分が分かるように書く。
  • 3. コピペで使える hook 例
    毎月 1-2 個、その月の事故から逆算した PreToolUse / Stop hook を出す。動作確認済みの bash スクリプトと、適用方法を添える。
  • 4. 失敗事例 1 件の深掘り
    月で最も影響の大きかった事故 1 件を、症状・原因・復旧手順・予防策・月次チェック項目の追加 まで分解する。GitHub Issue で「同じ症状」と言われている関連報告も束ねる。
  • 5. 既存商品の更新案内
    Token Book、Migration Playbook、Postmortems 本、Survival Guide、cc-safe-setup の各商品で、その月にどんな更新が入ったかを整理する。読み返すべき章があればその案内も書く。

なぜ月額で届けるか（一回の本との違い）

1. 根拠が古びない前提の商品

一回の本（移行の手引き、 事故報告本）は公開時の根拠を凍らせた文書。 5/6 朝に移行の手引きの第 11 触媒の章を 8 時間前の起稿状態と比較したところ、 18 件の引用のうち 4 件が既に閉じられていた（22%）。 8 時間で 22% の根拠が動いた領域では、 月次更新を前提にした商品が一回の本を上回る形になる。 利用者の集合的な認識の側でも、 公式の長期の約束と短期の実態の食い違いは 5/7 朝に主要な海外の議論の場で 1361 票の規模で共有された。 5/7 朝の別の海外の議論の場には、 公式の値段の調整への反発と、 中国の事業者の 75 パーセント引き（5/31 まで、 OpenAI 互換と Anthropic 互換の両方で接続可能）の発表が並列で 1 時間に 62 ポイントの反応で出ている。 動きの速度は月単位の納品物の必要性を直接示す。 月額の購読者は 6 月号、 7 月号と連続で受け取り、 過去の章で扱った Issue の続報も毎月反映される。

2. 数百件の Issue を月次で 9 件に圧縮

5 月号の第 1 部だけで、 8 日間（4/25-5/2）の公開 Issue、 利用者の集まり、 海外の場、 自分の運用記録を巡回し、 9 件の事故の集まりに整理した。 各集まりに該当の Issue 番号（空のキャッシュの印の問題だけで 14 件）、 発生のきっかけ、 対処の方向を入れた。 利用者が同じ整理を自分でやると数百件の Issue 巡回が必要。 月 ¥500 はこの巡回と整理を月次で外注する値段。

3. 月ごとの失敗様式に対応した動作確認済みの設定

5 月号の hook は当月の事故から逆算した動作確認済みの設定。 費用監視 hook は Issue #55445（事前のキャッシュの作成の量が 2 倍に膨らんだ退行）に直接対応する 1 日累計の閾値警告。 圧縮直前の保存 hook は #42817（自動の圧縮が効かない）と #55468（圧縮が通信不良で途切れる）の両方への保険。 設定例自体は再利用可能で、 過去号のものも蓄積する。 12 ヶ月の購読で 12-24 件の hook が手元に集まる。

Founder 価格について

初期メンバー向けの価格として ¥500/月 で固定する。 あとから価格を上げても、 Founder で参加した会員はそのまま据え置きにする（grandfathered）。 たとえば将来 ¥1,000/月 や ¥1,500/月 に値上げしても、 5 月のうちに Founder で参加した会員は ¥500/月 のまま継続できる。 解約しない限り、 一度確定した値段が変わることはない。 Max plan の事故 1 件で 1 年分以上の元が取れる前提で設計している。

これは何ではないか（誠実な開示）

含まれないもの：

  • 1 対 1 のサポートやチャット相談（コンテンツ商品としての提供）
  • 専用 Discord や Slack（個別の議論場は用意していない）
  • Anthropic 公式との特別な連携や事前情報（一次情報は誰でも見られる Issue を整理する立場）
  • 「絶対に事故が起きなくなる」保証（事故を完全に防ぐ仕組みではなく、起きたものを早く知り、対処を共有するもの）

合うかどうかの目安： Claude Code を週 5 日以上業務で使っており、毎月の事故と仕様変更を 30 分以内で把握したい人には合う。月 1 回しか触らない人や、事故が起きても影響が小さい使い方の人には合わない。

Ko-fi での配信の仕組み

Ko-fi は、 個人の作者が月額の購読や 1 回限りの応援を集めるための海外のサービスで、 Stripe でのカード決済に対応している。 配信の仕組みは次の 4 段階。

  1. 登録： Ko-fi のアカウントを作る (メールアドレスのみで可能)。 既存の Google アカウントでもログイン可能。
  2. 支払い： Founder の階層を選び、 Stripe でカード情報を入力する。 ¥500/月 の自動課金が始まる。 日本円のカードでも米ドルのカードでも対応。
  3. 受け取り： 月 1 回、 Ko-fi のメンバー限定投稿として PDF または Markdown が公開される。 投稿の通知はメールで届く (Ko-fi のメールアドレスへ)。
  4. 読み方： 通知のリンクから Ko-fi のメンバー限定の頁にアクセスし、 ファイルをダウンロードする。 ダウンロード後はオフラインでも読める。 過去の自分の参加月以降の号は何度でも再ダウンロード可能。

解約はいつでも Ko-fi の管理画面から可能。 解約の翌月から請求は止まる。 日割り返金は Ko-fi の仕様で対応していない。

他の選択肢との違い

Claude Code 関連の月額の有料媒体は英語圏に複数ある（読者 20 万〜38 万人規模）。 ただし、 GitHub Issue から事故事例を月次で構造化して届ける形式の月額媒体は、 私が調べた範囲では英語圏にも日本語圏にも存在しない。 既存の英語圏の媒体は週次の戦略・ 採用・ 運用の読み物で、 事故事例の継続的な追跡を主軸にしていない。 主要な英語圏の媒体と 4 つの軸で比較する。

単発の解説記事との違い：3/31 の流出事件のような大きな事案は、セキュリティ企業（SQAT、TrendMicro）、大手 SI（サーバーワークス）、主要技術媒体（@IT、SBBit）、個人開発者（Zenn、Qiita）が単発の解説記事を出す。これらは事件の全体像と対策をその時点で押さえるには有効だが、翌月以降に新しい事故が起きた時の続報の経路がない。Safety Lab は「事件の後追い」 ではなく「日々の運用の更新」 を月次で続ける位置づけ。中小企業の経営層向けの予防啓発（無料領域）とも、単発解説（個別記事）とも重ならない、個人開発者の月次実践のスロットを埋める。

2026-05 Failure case: cache_control on empty text blocks jams sessions completely

What happens

Even short sessions suddenly start returning 400 errors and work stops. The two messages that typically appear, in this order:

messages.N.content.M.text: cache_control cannot be set for empty text blocks
messages: text content blocks must be non-empty

"Try again" and claude --resume both fail with the same error. Starting a fresh session works but discards all the context.

Why it jams

Somewhere in the conversation history, a content block is left as text: "" with cache_control: { type: "ephemeral" } attached, and the API rejects it. Once the broken block is in history, every retry includes it, so resume and Try again fail for the same reason.

Even if "the session feels short", every tool call adds two messages (assistant tool_use and user tool_result). Tool-heavy sessions reach messages[48] in just a few turns.

When it occurs (from the 2026 April-May cluster)

  • Right after pasting an image without a caption
  • Auto-resend after a stream wait timeout
  • When AskUserQuestion replies are submitted empty
  • Right after resuming a session whose history was already broken

Across GitHub Issues, the same root cause spawned more than 12 separate threads. Before publication, each thread was checked and only those whose error text and reproduction steps match were kept: #55369 / #55156 / #55302 / #55283 / #55118 / #54988 / #54421 / #54314 / #53632 / #50738 / #50681 / #50010 and others.

The remaining ~2,500 characters cover what the broken line looks like, the one-liner grep to detect it, the Python recovery script, three prevention procedures, and a monthly check item for your environment.
